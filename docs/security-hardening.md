# Security Hardening Notes

## Clinic QR Model

Each clinic should have its own patient intake QR code. The platform admin console generates these
tokens and renders downloadable/printable QR images; the SQL below documents the same flow for
operators.

The QR must not contain `clinic_id`, `hospital_name`, doctor name, or other tenant identifiers as query parameters. Use an opaque, rotatable intake token generated by the database:

```sql
select *
from public.create_clinic_intake_token(
  '<clinic uuid>'::uuid,
  'Front desk QR',
  null
);
```

Use the returned `token` only in the QR URL fragment:

```text
https://your-patient-form.example/#/i/<token>
```

Fragments are not sent to web servers in HTTP requests, which reduces leakage through CDN, proxy, browser history sync, analytics, and access logs. The browser submits the token in the form body, and WF11 resolves it server-side through `public.resolve_public_intake_token`.

The patient form also accepts a path-based form (`https://your-patient-form.example/i/<token>`) for
deployments that prefer a clean URL; the `patient-form/vercel.json` rewrite serves the SPA for
`/i/*`. The hash form is preferred where token leakage through access logs is a concern.

Rotate a QR by disabling the old token and creating a new one:

```sql
update public.clinic_intake_tokens
set status = 'disabled'
where clinic_id = '<clinic uuid>'::uuid
  and label = 'Front desk QR';
```

## Admin Console

The platform admin console (`admin-console/index.html`) is a standalone static page used to
review onboarded clinics, mint per-clinic QR tokens, seed/clear demo patients, and view any
clinic's queue. It is backed by Supabase and gated end to end.

Clinic registration is **not** performed in the console. Clinics self-register through the
hospital-onboarding form (WF12), which calls `get_or_create_clinic_id` and writes the
`hospital_boarding` row; onboarded clinics then surface in the console automatically. The console
is read-only for clinic data and exposes `admin_list_clinics` / `admin_get_clinic_details` for
review (contact, doctors, registration #, QR/scan activity, and a "needs attention" flag for
clinics that still lack an active QR or a doctor WhatsApp number).

### Authentication and authorization

- Admins sign in with **email + password** (Supabase Auth, separate from doctor WhatsApp OTP).
- Immediately after sign-in the page calls `public.current_user_is_platform_admin()`; a session
  that is not a platform admin is signed out and denied. The client check is cosmetic — every
  privileged operation is a `SECURITY DEFINER` RPC that re-asserts
  `current_user_is_platform_admin()` server-side, so a forged UI cannot bypass it.
- Platform admin status comes from `public.platform_admins` (decoupled from any single clinic) or
  a `super_admin` clinic membership. There is **no** insert/update policy on `platform_admins`; the
  roster is changed only via SQL or the service role.

### Bootstrapping the first admin

1. Create the admin auth user (Supabase Dashboard → Authentication → Add user, with a strong
   password; enable email confirmation).
2. Grant platform admin in SQL:

```sql
insert into public.platform_admins (user_id, label)
values ('<auth user uuid>', 'Founder admin')
on conflict (user_id) do nothing;
```

3. Apply migrations (`npm run preflight`) so the admin RPCs and policies exist.

### Read access model

`schemas/migration-admin-console.sql` adds **read-only** platform-admin RLS policies
(`FOR SELECT` only) to `patients`, `patient_visits`, `prescriptions`, `prescription_medicines`,
`doctor_profiles`, `hospital_boarding`, `message_logs`, `message_ledger`, and `system_logs`.
Writes are never broadened by RLS; all admin writes go through the gated RPCs
(`create_clinic_intake_token`, `admin_set_token_status`, `admin_seed_dummy_patients`,
`admin_clear_dummy_patients`). Clinic creation happens only via the onboarding form, not the
console.

### Demo patients

- Seeded patients/visits/prescriptions are tagged `is_demo = true` and use the unassigned `+999`
  country code so they can never deliver a real WhatsApp message.
- The scheduled messaging workflows (WF1–WF5) additionally exclude
  `COALESCE(is_demo, FALSE) = FALSE`, so demo data never enters any outbound send path.
- Demo prescriptions are kept as drafts so `admin_clear_dummy_patients` can delete them; issued
  prescriptions are intentionally immutable and undeletable.
- Cleanup is clinic-scoped and only removes `is_demo` rows — real patients are never affected.

## Risk Controls

- Public patient intake no longer fetches the global clinic directory.
- Public patient intake no longer submits raw `clinic_id`.
- WF11 rejects missing, malformed, expired, disabled, or wrong-doctor intake tokens.
- Dashboard Supabase configuration is no longer accepted from URL query parameters.
- Hospital onboarding no longer accepts webhook or clinic overrides from URL query parameters.
- Prescription PDF links are expiring and HMAC-bound to prescription id, clinic id, and expiry.

## Remaining Operational Requirements

- Keep `TWILIO_VALIDATE_WEBHOOK_SIGNATURE=true` in production.
- Keep `INTERNAL_WEBHOOK_SECRET` high entropy and shared only by trusted server contexts.
- Do not expose service-role keys in static files or browser config.
- Prefer an authenticated admin-only onboarding flow before using hospital onboarding for production tenant creation.
- Do not publish QR token values anywhere except the QR asset itself and trusted clinic operations records.
- Make browser form origins and webhook origins compatible. A static form hosted on a different domain from n8n must use either a same-origin proxy/gateway or n8n/proxy CORS headers that allow the exact deployed form origin. A working `curl` request is not enough; browsers require `Access-Control-Allow-Origin` on the webhook response.
